## 一、项目愿景与核心价值

### 1.1 一句话定位
**让想法从"灵光一现"到"落地成真"的 AI 助理**

### 1.2 核心价值主张

| 维度 | 传统方式 | 「破」的方式 |
|------|----------|--------------|
| 想法管理 | 记在脑子里/笔记里，容易忘 | 统一入库，定期盘活 |
| 评估判断 | 凭直觉，容易高估/低估 | 结构化评估，减少盲区 |
| 执行追踪 | 开始轰轰烈，后来不了了之 | 任务级追踪，有始有终 |
| 复盘迭代 | 做完了就完了 | 定期复盘，持续优化 |

### 1.3 核心原则：实事求是

> **原则来源**：用户的核心价值观——"实事求是是第一原则，从第一性原理判断可行性"

```
┌─────────────────────────────────────────────────────────┐
│                    实事求是原则                          │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  1. 说可行性，也说局限性                                │
│     → 评估不仅给出"能做成什么"，也指出"可能遇到什么问题" │
│                                                         │
│  2. 说优势，也说风险                                    │
│     → 全面分析，不只报喜不报忧                          │
│                                                         │
│  3. 说可能性，也说约束条件                              │
│     → 客观评估，不画大饼                                │
│                                                         │
│  4. 承认不知道                                          │
│     → 不确定的地方明确说出来，不胡乱猜测                 │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

**在「破」中的应用**：

| 模块 | 应用场景 |
|------|----------|
| 深度评估 | 不仅评估"创新性/可行性/价值性"，也评估"风险/局限/约束" |
| 任务拆解 | 诚实预估时间和难度，不夸大 |
| 复盘机制 | 真实记录失败原因，不回避问题 |
| 报告生成 | 双向反馈：成功经验 + 失败教训 |

---

### 1.4 目标用户画像

- **王晋**，跨学科研究者
- 特点：有大量工作/副业/学习想法，但精力有限，容易遗忘
- 诉求：不想错过好想法，把有限精力放在最高价值的任务上
- 期望：客观真实的评估，不被"画大饼"式的分析误导

## 三、评估标准体系

### 3.1 快速评估（分钟级）

**目的**：理解用户意图，过滤无效输入，提取关键信息

| 维度 | 检查项 | 评分标准 |
|------|--------|----------|
| 意图识别 | 是想法还是命令？ | 命令→直接执行，是想法→进入想法处理流程 |
| 关键词提取 | 识别领域标签 | 消防/副业/学习/技术/生活 等 |
| 完整性 | 是否有核心信息 | 有明确主题=0.8+，有具体描述=1.0 |

**快速评估输出示例**：

```
💭 想法录入成功

📋 快速评估结果：
• 类型：想法
• 关键词：副业、电商
• 完整性：0.8（建议补充：启动资金、预计收益）
• 状态：已加入想法库，等待深度评估

📝 后续操作：
• 「查看想法」- 查看想法列表
• 「立即评估」- 触发深度评估
```

### 3.2 深度评估（周级）

**目的**：全面评估想法的创新性、可行性、价值性，辅助用户决策

#### 3.2.1 创新性评估 (0-100分)

| 维度 | 检查项 | 权重 | 评分标准 |
|------|--------|------|----------|
| 新颖度 | 是否是全新概念 | 30% | 完全新=100，部分参考=60，借鉴为主=30 |
| 差异化 | 与现有方案的差异 | 25% | 有明显差异=100，有细微差异=60，模仿=20 |
| 解决问题的程度 | 能解决什么问题 | 25% | 核心问题=100，边缘问题=60，周边问题=30 |
| 趋势契合度 | 符合行业发展趋势 | 20% | 引领趋势=100，顺应趋势=70，逆趋势=20 |

**创新性评估 Prompt 示例**：

```
你是创新性评估专家。请评估以下想法的创新性（0-100分）：

想法：「基于物联网的智能消防巡检系统」

请从以下维度打分：
1. 新颖度（30%）：这个想法有多新？
2. 差异化（25%）：与现有消防系统有什么不同？
3. 解决问题（25%）：解决了什么核心问题？
4. 趋势契合（20%）：符合消防行业智能化趋势吗？

输出格式：
- 各维度得分及理由
- 创新性总分
- 简要评估
```

#### 3.2.2 可行性评估 (0-100分)

| 维度 | 检查项 | 权重 | 评分标准 |
|------|--------|------|----------|
| 技术可行性 | 现有技术能否实现 | 25% | 技术成熟=100，需要研发=60，技术难点大=20 |
| 资源需求 | 人力/资金/时间 | 25% | 资源充足=100，需要调配=60，资源匮乏=20 |
| 法规合规 | 消防行业法规要求 | 20% | 完全合规=100，需要认证=60，合规困难=20 |
| 团队能力 | 当前团队能否胜任 | 15% | 完全能=100，部分需要学习=70，需要招人=40 |
| 风险可控性 | 主要风险是否可应对 | 15% | 风险低=100，中等风险=60，高风险=20 |

#### 3.2.3 价值性评估 (0-100分)

| 维度 | 检查项 | 权重 | 评分标准 |
|------|--------|------|----------|
| 商业价值 | 能带来多少收益 | 25% | 高收益=100，中等=60，低=20 |
| 战略价值 | 对长期发展的意义 | 20% | 战略级=100，重要=70，一般=40 |
| 用户价值 | 对目标用户的帮助 | 25% | 强需求=100，弱需求=60，可有可无=20 |
| 差异化价值 | 能否建立竞争壁垒 | 15% | 能建立=100，部分能=60，不能=20 |
| 可扩展性 | 未来能否扩展 | 15% | 扩展性强=100，有限扩展=60，封闭=20 |

#### 3.2.4 风险与局限评估 (0-100分)

> **原则**：实事求是，不仅说"好"的部分，也要指出潜在问题和约束

| 维度 | 检查项 | 权重 | 评分标准（风险低/局限少=高分） |
|------|--------|------|--------------------------------|
| 主要风险 | 识别Top3风险 | 25% | 风险可控=100，有办法应对=70，难应对=30 |
| 已知局限 | 坦诚列出想法的局限性 | 25% | 局限少=100，有一些=60，较多=30 |
| 假设条件 | 这个想法依赖哪些假设？ | 20% | 假设少且容易验证=100，假设多=50，假设难验证=20 |
| 失败模式 | 什么情况下会失败？ | 15% | 失败模式清晰=100，部分清晰=60，不清晰=20 |
| 退出条件 | 什么时候应该放弃？ | 15% | 有明确退出点=100，模糊=60，无退出点=20 |

**风险与局限评估 Prompt 示例**：

```
你是风险评估专家。请客观分析以下想法的风险和局限性：

想法：「基于物联网的智能消防巡检系统」

请从以下维度分析（实事求是，不要只说好听的）：
1. 主要风险：列出最可能的3个风险点
2. 已知局限：这个想法本身有哪些局限性？（坦诚说）
3. 假设条件：这个想法依赖哪些假设？这些假设成立吗？
4. 失败模式：什么情况下这个想法会失败？
5. 退出条件：执行到什么程度如果还没成功应该考虑放弃？

输出格式：
- 各维度分析（每个维度50字以内）
- 风险评分（0-100）及理由
- 诚实建议：是否值得尝试？如果值得，什么情况下应该止损？
```

> **注意**：风险评估采用"实事求是"原则，不仅说好的部分，也要指出潜在问题。评估结果应包含"建议"和"警告"两部分。

#### 3.2.5 综合评分计算

```python
# 综合评分 = 前景评估 + 风险评估
# 前景评估（权重70%）
前景得分 = 创新性 × 40% + 可行性 × 35% + 价值性 × 25%

# 风险评估（权重30%，但提供独立参考）
风险得分 = 风险评估 × 100%（满分100）

# 最终综合评估
综合得分 = 前景得分 × 70% + 风险得分 × 30%

# 决策阈值（考虑风险调整后）
- 综合得分 ≥ 75分 且 风险得分 ≥ 60分：🌟 强烈推荐优先执行
- 综合得分 60-74分 或 风险得分 < 60分：⚠️ 谨慎推荐，需关注风险
- 综合得分 45-59分：📝 建议观望，持续关注
- 综合得分 < 45分 或 风险得分 < 40分：❌ 暂不推荐，优先考虑止损
```

**评估报告应包含**：

```
┌─────────────────────────────────────────────────────────┐
│                    评估结果摘要                           │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  前景评估（好的一面）：                                  │
│  ├── 创新性：XX分 - ...                                 │
│  ├── 可行性：XX分 - ...                                 │
│  └── 价值性：XX分 - ...                                 │
│                                                         │
│  风险评估（问题的一面）：                                │
│  ├── 主要风险：...                                      │
│  ├── 已知局限：...                                      │
│  ├── 假设条件：...                                      │
│  └── 失败模式：...                                      │
│                                                         │
│  诚实建议：                                             │
│  ├── 是否值得尝试？                                     │
│  ├── 如果尝试，预估成功概率？                           │
│  └── 什么时候应该止损？                                 │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 3.3 评估触发机制

#### 三种触发方式

| 触发方式 | 触发条件 | 优先级 | 说明 |
|----------|----------|--------|------|
| 定时触发 | 每周六 06:00 | 低 | 批量评估积累的想法 |
| 定量触发 | 每积累 5 个新想法（可调） | 中 | 避免想法积压太久 |
| 用户触发 | 用户发送「立即评估」 | 高 | 用户主动要求评估 |

#### 触发流程

```
触发条件满足
     │
     ▼
获取待评估列表
     │
     ▼
按综合优先级排序评估队列 ← 创新性+可行性+价值性综合排序
     │
     ▼
逐个深度评估（每次最多3个）
     │
     ▼
生成评估报告
     │
     ▼
推送用户确认
```

### 7.4 进度计算

```python
def calculate_progress(task: Task) -> int:
    """计算任务进度"""
    if not task.subtasks:
        return 100 if task.status == 'done' else 0
    
    done_count = sum(1 for s in task.subtasks if s['done'])
    return int(done_count / len(task.subtasks) * 100)

def calculate_idea_progress(idea: Idea) -> int:
    """计算想法整体进度"""
    if not idea.tasks:
        return 0
    
    total_progress = sum(t.progress for t in idea.tasks)
    return int(total_progress / len(idea.tasks))
```
